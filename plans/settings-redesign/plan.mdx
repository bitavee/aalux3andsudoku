<RichText id="objective">

### Objective & done criteria

Make all three settings surfaces feel like one polished app. The work is **consistency and hierarchy, not a new look** — the device runs a single hardcoded theme (`LyraTheme`, `UITheme.cpp:21`), and that theme already has a competent e-ink kit: rounded-rect selection with gray-dither fills, black "pills" for the focused value, 24/32px bitmap icons, header underline rules, and a real scrollbar. Two of the three surfaces simply don't use it.

**Done** =

- **Reader Menu** (`EpubReaderMenuActivity`) and **Quick Settings** (the `EpubReaderActivity` overlay) are rebuilt on the shared `GUI` primitives instead of hand-rolled hardcoded layouts — same header, list rows, selection, and value treatment as main Settings.
- **Main Settings** uses the kit's good parts it currently skips: within-tab section grouping, icon tiles on drill-in actions, and On/Off pills.
- The reader's two settings surfaces are **separated by purpose**: Reader Menu = navigation & actions; Quick Settings = live formatting.
- One value-editing model across all three (Up/Down moves, Left/Right steps the focused value, Confirm activates).
- Builds clean, host tests pass, and it reads correctly in all four orientations on device.

The four design forks are now resolved — see **Decisions** at the bottom.

</RichText>

<Callout id="decision" tone="decision">

**Approach: refine & extend Lyra, applied consistently.** Keep the in-progress tab-bar structure for main Settings; do not invent a new visual language. The biggest beauty-per-effort win is that the Reader Menu and Quick Settings hand-roll their own layout (black-inversion boxes at hardcoded Y positions like `15/45/75`) and look like a different, older app bolted on. Routing them through the same `drawHeader` / `drawList` / value-pill / glyph-hint primitives makes them consistent *and* deletes bespoke layout code. Flash is not a constraint here — the `default` image is **4.68 MB against the 6.55 MB limit (~1.87 MB free)**, RAM at 33% — so new icons and a couple of new primitives are affordable. The binding constraints are the e-ink panel (1-bit, slow refresh, grayscale is expensive) and RAM discipline.

</Callout>

<RichText id="surfaces-heading">

### The three surfaces

The mocks above are the target state; here is the per-surface intent and the technical delta.

**1 · Main Settings (`SettingsActivity`).** Keep the tab bar (Display / Reader / Controls / System). The Reader tab alone has ~14 items, so add light **section rules** (TEXT / LAYOUT / SCREEN) to break the wall without a second navigation level. Give **drill-in action rows** (Wi-Fi, File transfer, KOReader sync, Clear cache, Check for updates, Manage fonts, Status bar) a leading icon tile + trailing chevron so they read differently from inline value rows. Render toggles as iOS-style switches (a rounded track with a knob; knob-right + filled track = on) — see the legibility risk below. The focused-value black pill already works here (`drawList` is called with `highlightValue = true`); we only add the `‹ ›` stepper arrows. The bottom hint band uses the standard four-glyph `Navigation` set (◀ ● ▲ ▼), identical to the home screen — the in-progress `SettingsNav` variant (stacked up/down + left/right glyphs) is dropped. Confirm already cycles tabs and values (`SettingsActivity.cpp:91-98,153-165`), so Left/Right remain optional accelerators rather than earning their own glyphs.

**2 · Reader Menu (`EpubReaderMenuActivity`).** Today it's hand-rolled and mixes navigation, actions, and formatting in one flat 14-item list. Rebuild `render()` on `drawHeader` (book title + progress subtitle) + `drawList` (icon tiles, gray selection, chevrons). Strip it to **navigation & actions only**: Chapters, Bookmarks, Add bookmark, Go to position, Sync, Look-up, Screenshot, QR, Go home, Delete cache. A single **"Reading settings…"** row opens Quick Settings (the long-press gesture still opens it directly).

**3 · Quick Settings (`EpubReaderActivity` overlay).** Rebuild `renderQuickSettingsOverlay()` as a proper Lyra bottom sheet: rounded top corners, pill tabs (Reader / Controls), value rows with the focused stepper pill, toggle pills. It owns **all live formatting** — and gains **Orientation** and **Auto page turn** (both moved out of the Reader Menu). One deliberate divergence from the kit: selection is **solid black/white, not gray dither**, because the overlay intentionally skips the grayscale pass for instant nav (`EpubReaderActivity.cpp:1010`). A solid black pill is both fast and on-brand with Lyra's "black = active" idiom.

</RichText>

### Planned shape — data-driven grouping

To keep subheaders out of render logic, carry the group on each setting and let `drawList` emit a rule when the group changes. Smallest plausible shape:

struct SettingInfo {
  // ...existing fields...

<Callout id="risks" tone="risk">

**Constraints & risks.** (1) **Grayscale in the overlay** — Quick Settings must not trigger a grayscale pass over live text (`EpubReaderActivity.cpp:1010`); use solid black/white pills, verify nav stays instant. (2) **Four orientations** — the Reader Menu has bespoke landscape/inverted gutters today; moving to `drawList` must preserve correct layout in Portrait, Inverted, and both Landscapes (a known bug-magnet). (3) **Two paths must not drift** — Settings·Reader and Quick Settings duplicate the same formatting options; without a shared source they'll diverge — resolved by a single shared source of truth (see Decisions). (4) **Icon tiles** read as broken images if a bitmap is missing — every action row needs a real 24px icon before ship. (5) **RAM** — build all menu/list vectors in `onEnter()`, free in `onExit()`; no per-frame heap in the overlay render path. (6) **Toggle legibility** — the chosen iOS-style switch must stay unmistakable at ~20px on a 1-bit panel with no anti-aliased grays; verify on hardware and fall back to a labelled On/Off pill if the on/off states blur.

</Callout>
